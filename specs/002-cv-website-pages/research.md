# Research: Static CV Website Pages

**Feature**: 002-cv-website-pages | **Date**: 2026-04-28 (updated from 2026-04-13)

## R1: Next.js Static Export for Bilingual Site

**Decision**: Use Next.js 15 with `output: "export"` and App Router `[lang]` dynamic segment.
**Rationale**: Provides file-based routing, static generation, and React component model with zero server at runtime. The `generateStaticParams()` function in `app/[lang]/layout.tsx` pre-renders both language variants.
**Alternatives considered**:

- Vanilla HTML/CSS/JS: Would require hand-rolling SPA routing, language switching, search filtering, and expand/collapse — more code, more bugs, harder to maintain.
- Astro or 11ty: Viable but would introduce a new framework when Next.js is already in place and working.

## R2: Content Pipeline Architecture

**Decision**: Single TypeScript build tool (`tools/build-content.ts`) reads `specs-input/cv/` text files and outputs `public/content/site-content.json`. Pages consume this via static import at build time (`lib/content/loadContent.ts`).
**Rationale**: One generated JSON file keeps the runtime simple (no filesystem reads, no API calls). The build tool is TypeScript per constitution. The parser (`tools/content/parseSource.ts`) validates entry structure during build.
**Alternatives considered**:

- Per-page JSON files: More files to manage, no practical benefit for a site this small.
- MDX/Markdown: CV text format is already defined (spec 001); no need for a second content format.

## R3: UI Strings in site-content.json

**Decision**: All translatable UI strings (nav labels, section headings, search placeholder, "Show all", "present", "No results", form labels, not-found text) are stored in `site-content.json` under a `ui` key, generated by the build tool.
**Rationale**: Keeps all i18n in one place. Pages source labels from the same content object they use for CV data — no separate i18n library needed.
**Gap identified**: The JSON schema (`contracts/site-content.schema.json`) defines the `ui` field, but `build-content.ts` must generate it. This is a build tool task.

## R4: Visual Styling — Mockup as Source of Truth

**Decision**: The mockup `specs-input/mockup/scandinavian.html` is the canonical visual reference (FR-004b). The spec does not duplicate CSS values — implementers copy styles from the mockup.
**Rationale**: Avoids maintaining CSS in two places (spec + code). The mockup is a live HTML file that can be opened in a browser for pixel comparison.
**Design tokens** (from mockup `:root`): `--white`, `--off`, `--line`, `--text`, `--mid`, `--accent`, `--accent-light`, `--warm`.
**Typography**: Fraunces (serif, headings/name) + Instrument Sans (sans, body/nav/labels).

## R5: Contact Form — EmailJS Integration

**Decision**: Client-side email sending via `@emailjs/browser`. Three env vars (`NEXT_PUBLIC_EMAILJS_PUBLIC_KEY`, `_SERVICE_ID`, `_TEMPLATE_ID`) read at runtime.
**Rationale**: No backend needed — aligns with static-only constitution. EmailJS handles SMTP. Missing env vars trigger error state (FR-030) rather than unhandled exceptions.
**Alternatives considered**:

- Formspree/Netlify Forms: Would add a third-party form endpoint dependency. EmailJS is already integrated.
- Server function: Would require a backend, violating the static-only constitution.

## R6: Language Routing and Redirects

**Decision**: `[lang]` dynamic segment with `generateStaticParams()` for `en`/`sv`. Non-prefixed routes (`/about`, `/contact`, etc.) use Next.js `redirect()` to `/en/<route>`. Root `/` redirects to `/en/`.
**Rationale**: Clean URLs, no middleware needed, works with static export.
**404 limitation**: The not-found page has no language segment — it defaults to English (FR-031). This is intentional per spec.

## R7: CV List Interaction Patterns

**Decision**: Client-side search, expand/collapse, and show-all managed in `CvPageClient.tsx` + `CvSections.tsx`.

- **Search**: Multi-word AND match (case-insensitive) across period, title/role, organization, client, location, description.
- **Show-all**: Default 5 items; "Show all" reveals rest. Disabled during active search (all matches shown).
- **Expand/collapse**: Per-item state toggle on click.
  **Rationale**: All interactions are client-side — no API calls. Search is instant since all data is already in the page.

## R8: Technology Taxonomy and Years Computation

**Decision**: Build tool intersects `specs-input/tech-categories.json` with `Technologies:` fields from all CV entries. Only technologies appearing in at least one entry are included. Unmapped technologies go to "Other" (rendered last). For each technology, the build tool computes total years by summing the duration (months / 12, rounded) of every CV entry whose `Technologies:` field includes that tech. Ongoing entries use the build date as end. Technologies with 0 years (< 6 months total) are still included and displayed as "< 1 yr" per FR-037.
**Rationale**: Keeps the tech page data-driven — derived from CV entry periods rather than hardcoded values. Build-time computation means zero runtime cost. Constitution III compliance: `specs-input/cv/` remains the sole source of truth.

## R9: Technology Chart Visualization

**Decision**: The Technologies page renders an interactive chart with horizontal rows (name + progress bar + years label), category filter buttons, and a Time/A–Z sort toggle. This is a client component (`TechChart.tsx`) that manages filter and sort state.
**Rationale**: The mockup (`scandinavian.html`) implements this exact pattern — chart rows with progress bars rather than pill chips. FR-004b mandates matching the mockup. The chart conveys both breadth (number of technologies) and depth (years of experience) simultaneously.
**Progress bar scaling**: Width = `years / maxYears × 100%` where `maxYears` is the maximum across all technologies (no hardcoded constant). Color tiers: ≥12yrs `#1a6b4a`, ≥7yrs `#2d9b6f`, ≥3yrs `#6bbea0`, <3yrs `#a8d8c8`.
**Data shape change**: `technologies` in `site-content.json` changed from `Record<string, string[]>` to `Record<string, Array<{ name: string; years: number }>>`.
**Alternatives considered**:

- Pill chip tag cloud (original spec): Did not match the mockup. Showed breadth but not depth (no years information).
- Third-party chart library (Chart.js, Recharts): Overkill — the chart is simple enough with CSS + HTML divs.

## R10: Externalize All Hardcoded Strings (FR-038)

**Decision**: Extend the existing `ui` UIStrings structure in `site-content.json` with new paths for all remaining hardcoded strings. Components consume these via the same `loadContent()` pipeline already used for existing UI labels. No new i18n library or pattern is introduced.
**Rationale**: The UIStrings pattern (build-tool-generated `LocalizedText` objects under `ui.*`) is already proven across nav labels, CV section headings, search placeholder, and contact form labels. Extending it to cover the remaining hardcoded strings is the simplest path — same pattern, same data flow, same types.
**Strings identified**:

| Current location                          | Hardcoded string(s)                       | New UIStrings path                                              |
| ----------------------------------------- | ----------------------------------------- | --------------------------------------------------------------- |
| `app/layout.tsx`, `app/[lang]/layout.tsx` | "Malte Särner - CV", "Malte Särners CV"   | `ui.meta.title`, `ui.meta.description`                          |
| `app/not-found.tsx`                       | "Go to home page"                         | `ui.notFound.homeLink`                                          |
| `components/ContactForm.tsx`              | "This field is required" / sv             | `ui.contact.validation.required`                                |
| `components/ContactForm.tsx`              | "Please enter a valid email address" / sv | `ui.contact.validation.email`                                   |
| `components/ContactForm.tsx`              | "Sending…" / "Skickar…"                   | `ui.contact.sending`                                            |
| `components/TechChart.tsx`                | "All"                                     | `ui.technologies.allCategory`                                   |
| `components/TechChart.tsx`                | "Time"                                    | `ui.technologies.sortTime`                                      |
| `components/TechChart.tsx`                | "A–Z"                                     | `ui.technologies.sortAlpha`                                     |
| `components/TechChart.tsx`                | "< 1 yr", "1 yr", "yrs"                   | `ui.technologies.yearsLessThanOne`, `.yearsOne`, `.yearsSuffix` |

**Alternatives considered**:

- Separate i18n library (next-intl, react-i18next): Unnecessary overhead — the site has only 2 languages and ~30 string paths. The existing pattern works.
- Inline language maps in components: Would keep strings in source code, violating FR-038 and making translation management harder.

**Swedish translations for new paths** (confirmed):

- "A–Ö" (not "A–Z") — Swedish alphabet ends with Å, Ä, Ö
- "Alla" for "All", "Tid" for "Time"
- "< 1 år", "1 år", "år" for year labels

## R11: Technologies in CVEntry for Search, Description Paragraphs, About Page HTML

**Decision**: Three incremental changes to improve CV search, text rendering, and about page links.

### R11a: Add technologies[] to CVEntry JSON

**Decision**: Include the parsed `Technologies:` field from `meta.txt` as a `technologies: string[]` property on each `CVEntry` in `site-content.json`. The search haystack in `CvSections.tsx` includes this array (joined as a space-separated string).
**Rationale**: The `Technologies:` field is already parsed by `build-content.ts` for the aggregated tech chart. Currently it's discarded from individual CVEntry objects. Adding it to the JSON output enables searching for "React" or "TypeScript" to find matching assignments. This aligns with User Story 2b acceptance scenario 2.
**Alternatives considered**:

- Full-text index: Overkill for a static site with ~50 entries. Simple string inclusion is sufficient.
- Embed tech names in description text: Would create duplication vs. the structured `Technologies:` metadata.

### R11b: Render description lines as separate paragraphs

**Decision**: In the expanded CV entry view (`CvSections.tsx`), split `description[lang]` on `\n`, filter out empty lines, and render each line as a separate `<p>` element with vertical spacing (`space-y-3`).
**Rationale**: Source text files contain paragraph breaks as newlines. React renders `{text}` as a single text node, collapsing all whitespace. Splitting and mapping to `<p>` elements preserves the intended paragraph structure. This implements FR-024's updated requirement.
**Alternatives considered**:

- CSS `white-space: pre-wrap`: Would preserve blank lines as extra vertical space, which the spec says to omit.
- Markdown rendering: Overkill — the content is plain text paragraphs, not formatted markdown.

### R11c: About page HTML rendering

**Decision**: Use `dangerouslySetInnerHTML` for the about text in `app/[lang]/about/page.tsx` so that embedded `<a>` tags render as clickable links.
**Rationale**: The about text in `specs-input/about/{en,sv}.txt` contains HTML anchor tags for the GitHub repository link. React escapes HTML by default when using `{text}`. Since this content comes from static, developer-controlled source files (not user input), `dangerouslySetInnerHTML` is safe.
**Alternatives considered**:

- Custom link parser (regex to detect URLs): More complex, fragile, and the source already uses HTML tags.
- Markdown rendering: Would require changing the source format and adding a markdown library.

## R12: Responsive Mobile Layout (FR-040–FR-052)

**Decision**: Add a CSS media-query breakpoint at `max-width: 720px` following the updated mockup. Major mobile adaptations: hamburger menu, summary truncation, stacked card layout, mobile tech filter nav, and sr-only H1 titles. Implemented via Tailwind responsive utilities and a reusable `HamburgerDrawer` component.

### R12a: Reusable HamburgerDrawer component

**Decision**: Extract a `HamburgerDrawer` client component that provides: a hamburger button (three `<span>` lines with CSS transform animation to X), an active label, and a slide-down drawer container accepting children. Used for both main nav and tech filter nav.
**Rationale**: The mockup uses the exact same hamburger+drawer pattern for both the main navigation and the technology page filters. Extracting a shared component avoids duplication and ensures consistent interaction behavior (open/close animation, touch targets, overlay styling).
**Alternatives considered**:

- Headless UI dialog/popover: Adds a dependency for something achievable with a few lines of state + CSS transition.
- Separate hamburger implementations per location: Would duplicate ~50 lines of identical UI code.

### R12b: Summary truncation on mobile

**Decision**: Use CSS `-webkit-line-clamp: 3` with `overflow: hidden` on the summary paragraph, controlled by a `truncated` CSS class. A "Show all" button toggles the class off (one-way expand). On desktop the class is not applied.
**Rationale**: CSS line-clamp is widely supported (all modern browsers including Safari) and avoids JavaScript text measurement. One-way expand matches the mockup behavior (no re-collapse).
**Alternatives considered**:

- JS-based text truncation (character count): Fragile with variable font sizes and widths; CSS line-clamp is more robust.
- `text-overflow: ellipsis`: Only works for single-line truncation, not multi-line.

### R12c: sr-only H1 titles

**Decision**: All page H1 elements use a `sr-only` CSS class (screen-reader-only, visually hidden) so they remain in the DOM for SEO/accessibility but don't consume visual space. The mockup hides them in both desktop and mobile layouts.
**Rationale**: SEO best practice requires one `h1` per page. The mockup's visual design doesn't use visible page headings (the nav and content provide sufficient context). `sr-only` is the standard pattern used by Tailwind and Bootstrap for this purpose.

### R12d: Stacked periods on mobile

**Decision**: On mobile (≤720 px), CV role cards and assignment rows use `flex-wrap: wrap` with the period element set to `flex-basis: 100%; order: -1` so it renders above the title instead of to the right.
**Rationale**: On narrow screens, inline title+period doesn't fit. The mockup stacks period above title. Using CSS `order` and `flex-wrap` achieves this without changing the HTML structure or component logic.

### R12e: Mobile tech filter nav

**Decision**: On mobile, the desktop tech filter bar (category buttons + sort toggle) is hidden via CSS. A separate mobile filter bar appears using the `HamburgerDrawer` component, with sort toggle and category buttons inside the drawer.
**Rationale**: The desktop filter bar with many category buttons wraps poorly on narrow screens. The mockup solves this with a hamburger drawer pattern identical to the main nav. Reusing `HamburgerDrawer` keeps both consistent.
